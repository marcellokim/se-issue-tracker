# 워크플로우 예외 상황 정리 (Workflow Exception Cases)

> 이 문서는 ITS의 정상 워크플로우(PDF §2.2 happy path) 외에 발생할 수 있는
> 예외 상황을 식별하고, 각 상황에 대한 처리 방침 후보를 정리한 문서입니다.
>
> 각 상황의 처리 방침이 확정되면 `assumptions.md` 또는 관련 명세 문서에 반영합니다.

---

## 상황 1: 이슈 종속성

**시나리오:** User가 이슈 C를 등록했는데, 분석 결과 *"C를 해결하려면 B를 먼저 해결해야 하고, B는 A에 의존한다"* 가 발견됨.

**처리 방침:**
- 가정 사항 추가로 제외할지 아니면 기능적으로 구현할지 결정

---

## 상황 2: 이슈 해결 중 추가 이슈 발견

**시나리오:** Dev가 이슈 A를 해결하려다 추가적인 이슈 B를 발견. A를 해결하긴 했지만 B는 미해결 상태로 남겨둠.

**처리 방침:**
- 모든 User가 이슈 등록을 할 수는 있어서 문제 없긴 하지만, 새로 발견된 이슈 B가 종속성 이슈라면 상황 1의 문제가 발생

---

## 상황 3: 잘못된 fixed 처리

**시나리오:** Dev가 Fixed로 전이했지만 Tester가 검증 시 *"실제로는 안 고쳐졌다"* 고 판단하여 Resolved로 전이시키지 않음.

**처리 방침:**
- **옵션 A:** 이슈를 등록한 USER에게 FIXED → ASSIGNED 역전이 권한 부여 (e.g. False Fixed 버튼 누를 시 issue 상태는 다시 Assigned로 되돌아가고 Dev에게 알림 전송), 같은 dev에게 다시 배정된 상태로 돌림
- **옵션 B:** 일단 Resolve 상태 전이는 시키되, 잘못된 Fixed 상태였음을 PL에게 알리는 별도 기능 추가해서 PL이 해당 이슈를 일단 Close 시키고 적당한 assignee를 재배정 후 Reopen 시킴
- 가정 사항 추가로 제외도 가능은 함

---

## 상황 4: 이슈 중복

**시나리오:** 이슈 등록 시점에 이미 같은 내용의 이슈가 존재함.

**처리 방침:**
- 시스템이 자동으로 이슈 중복을 검출하지는 않음
- PL이 이슈 브라우즈 중 중복되는 이슈를 발견 시 나중에 등록된 중복 이슈에 코멘트를 남기고 CLOSED로 전이
- DUPLICATED 상태 추가 여부 결정 필요

---

## 상황 5: Assignee 변경 (재배정)

**시나리오:** PL이 dev1을 assignee로 지정했지만 dev1이 진행이 불가능하다고 PL에게 알리고, PL이 다른 적절한 dev에게 이슈 할당.

**처리 방침:**
- Assignee 재배정 기능 추가, ASSIGNED 상태에서 PL이 assignee 변경 가능

---

## 상황 6: Assignee 추가

**시나리오:** PL이 dev1에게 특정 이슈에 배정한 상태에서 dev1이 PL에 추가 인력 투입 요청.

**처리 방침:**
- ASSIGNED 상태에서 PL이 assignee 추가

---

## 상황 7: Priority 변경

**시나리오:** 이슈 등록자가 minor로 등록했는데 PL이 *"이건 사실 critical"* 이라고 판단.

**처리 방침:**
- **옵션 A:** 변경 불가, 등록 시 결정된 priority 고정
- **옵션 B:** PL만 우선순위 변경 가능, 트리아지(triage) 권한
- **옵션 C:** PL과 assignee 모두 변경 가능

---

## 상황 8: 등록 후 Title/Description 수정 가능 여부

**시나리오:** 이슈 등록자가 이슈를 등록했는데 오타 발견 또는 정보 추가 필요.

**처리 방침:**
- **옵션 A:** 수정 불가, 등록 후 이슈는 immutable, 정정은 코멘트로만
- **옵션 B:** reporter만 수정 가능, 자기가 등록한 이슈만 수정
