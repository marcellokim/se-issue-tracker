# Assumptions

이 문서는 텀 프로젝트 초기에 합의한 기본 가정을 정리합니다. 구현이 진행되면서 바뀌면 PR과 함께 갱신합니다.

## 상태 전이 정의
기본 이슈 상태 흐름은 다음과 같이 가정합니다.

`Backlog -> Ready -> In Progress -> In Review -> Done`

보조 규칙:
- 구현 전에는 반드시 Issue를 생성합니다.
- PR이 열리면 기본 상태는 `In Review`로 이동합니다.
- PR이 머지되고 관련 이슈가 닫히면 `Done`으로 이동합니다.

## 저장소 방식 (File Persistence)
초기 버전에서는 파일 기반 persistence를 사용한다고 가정합니다.

- 로컬 파일을 통해 데이터 저장
- DB 도입 전까지는 직렬화/역직렬화 가능한 구조 우선
- 경로, 포맷, 예외 처리 정책은 추후 별도 문서화

## UI 툴킷
초기 가정 UI 구성은 다음과 같습니다.

- 사용자용 메인 UI: JavaFX
- 관리자/보조 도구 UI: Swing

혼합 사용 시 공통 도메인 모델과 서비스 계층은 UI 프레임워크와 분리합니다.

## 추천 방식 (Heuristic)
추천 기능은 초기에는 heuristic 기반으로 구현한다고 가정합니다.

예시 기준:
- 우선순위
- 최근 활동 여부
- 상태
- 담당 가능성 또는 관련 태그

초기 단계에서는 explainable rule-based 접근을 우선하고, 추후 필요 시 개선합니다.

## 권한 가정
초기 권한 모델은 단순화합니다.

- 일반 사용자
- 관리자

필요 시 아래 항목을 확장합니다.
- 생성/수정/삭제 권한
- 상태 변경 권한
- 관리자 전용 화면 접근 권한
