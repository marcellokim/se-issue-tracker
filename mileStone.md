## M0: 과제 핵심 포인트 추출 (~4/30)

과제 명세를 읽고, 뭘 만들어야 하는지 각자 머릿속에 정리하는 단계.

### 구현 필수 기능
- [ ] 계정 관리 (admin, PL, dev, tester 역할 구분)
- [ ] 프로젝트 추가/관리
- [ ] 이슈 CRUD (Title, Description, Reporter, Reported Date 등 필수 필드)
- [ ] 이슈 브라우즈 & 검색 (assignee, 상태, reporter 등 필터)
- [ ] 이슈 상태 전이 (new → assigned → fixed → resolved → closed / reopened)
- [ ] 이슈 코멘트 추가 (누적, 날짜 포함)
- [ ] 이슈 상세 정보 확인 (필드 전체 + 코멘트 히스토리)
- [ ] 이슈 통계 분석 (일/월별 발생 횟수, 트렌드)
- [ ] 데이터 영속성 (DB든 파일이든, 재시작해도 데이터 유지)

### 설계 제약
- [ ] MVC 아키텍처 — UI와 로직 완전 분리
- [ ] UI 2개 이상 (Swing + JavaFX 등), 모델 코드 수정 없이 UI만 교체 가능해야 함
- [ ] JUnit 테스트 — 모델 레이어 대상

### 추가 기능 (가산점)
- [ ] assignee 자동 추천 — resolved/closed 이슈 이력 기반, 후보 3명 순서대로 제시
- [ ] "learn now" 버튼 — 데모용으로 수동 학습 트리거

### 토론 필요한 부분
- 추천 알고리즘 방식 결정 (TF-IDF 유사도? 단순 빈도 기반? ML?)
- 영속성 방식 결정 (SQLite vs JSON 파일 vs H2 등)
- UI 2개 중 어떤 조합으로 갈지 (Swing+JavaFX가 무난)
- 이슈 상태 전이 규칙 — 누가 어떤 상태로 바꿀 수 있는지 권한 매트릭스

## 참고: 이슈 상태 전이 흐름

```
new → assigned (PL이 assignee 지정)
    → fixed (dev가 수정 완료)
    → resolved (tester가 확인)
    → closed (PL이 종료)

어디서든 → reopened 가능 (필요시)
```

## 참고: 주의사항 요약

- **AI 활용 흔적 감점** — 문서/코드에서 AI 냄새 나면 감점. Q&A에서도 체크함.
- **설계 > 구현** — 돌아가기만 하면 되는 게 아님. 왜 이렇게 설계했는지 설명할 수 있어야 함.
- **GitHub 히스토리** — 커밋 로그가 개발 과정을 잘 보여줘야 함. 막판 몰아치기 티 나면 안 좋음.
- **UI 교체 데모** — 컴파일 & 빌드를 직접 보여줘야 함. 스크린샷만으로는 부족.
